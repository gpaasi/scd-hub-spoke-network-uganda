## Summary

This repository supports the paper, **“Linking newborn screening to essential sickle cell disease care in Uganda: a spatial hub-and-spoke network analysis of access, readiness, and strengthening priorities.”** The repository contains the analysis outputs, manuscript-linked supplementary materials, map figures, metadata, and validation scripts for a national newborn screening-to-sickle cell disease (SCD) care network in Uganda.

The analysis links accepted geocoded newborn screening (NBS) facilities to eligible Health Centre IV-and-above SCD care hubs. It evaluates whether screen-positive infants can be linked to geographically plausible and service-ready receiving facilities, and summarises route feasibility, hub readiness, hub workload, weak linkages, service gaps, and facility strengthening priorities.

The workflow is designed to support implementation planning for a screening-to-care pathway, including:

- newborn screening linkage
- specialised SCD clinic follow-up
- hydroxyurea access
- transfusion availability or escalation support
- hub workload monitoring
- staged strengthening of weak routes and partial hubs

It moves from facility-layer construction and service-readiness classification to spoke-to-hub distance matrices, nearest eligible hub assignment, route-feasibility classification, hub workload and route-risk summaries, facility-strengthening tiers, service-gap assessment, manuscript-ready maps, supplementary tables, and reproducibility checks.

Key outputs include cleaned and derived CSV files, route assignment tables, scenario-level distance summaries, hub readiness and workload summaries, strengthening priority tables, service-gap summaries, map figures, supplementary methods, supplementary tables, and validation reports.

## Important interpretation note

This repository supports a planning-oriented spatial network analysis. The assigned hub-and-spoke routes should be interpreted as modelled screening-to-care linkages, not as observed referral behaviour, patient movement, transport routes, or road-network travel times. Straight-line distances are used as first-pass planning distances. They do not capture road quality, terrain, seasonality, public transport, travel cost, caregiver preference, facility bypassing, or referral completion.

Hub readiness indicators represent measured availability of selected SCD service domains, not verified quality, reliability, staffing, stock continuity, blood availability, prescribing capacity, or effective coverage. Strengthening tiers identify facilities for programme review, validation, strengthening, or monitoring. They are not final implementation decisions.

## Run the validation checks

The repository includes a validation script that checks core manuscript results against the processed CSV outputs.

From the repository root, run:

```bash
python scripts/validate_results.py
```

The script writes validation outputs to:

```text
outputs/validation/validation_report.md
outputs/validation/validation_report.csv
```

The validation checks cover the major manuscript result domains, including accepted NBS spokes, candidate hubs, eligible hub counts, primary assigned routes, assigned distance bands, route feasibility, weak or long routes, hub workload, route risk, strengthening tiers, and service gaps.

## Reproducibility

This repository is organised as a reproducibility and audit package. The processed CSV files are the source-of-truth outputs for the manuscript results. The validation script checks whether the manuscript-level values are internally consistent with those outputs.

For reproducibility:

- keep processed analysis outputs in `data/processed/`
- avoid manually editing derived CSV outputs
- document any replacement or rerun of analysis files
- use `data/metadata/results_crosswalk.csv` to trace manuscript claims to source files
- rerun `scripts/validate_results.py` after changing any processed result file
- keep manuscript and supplementary files versioned under `docs/`
- treat route assignments as modelled potential linkages unless observed referral data are added

## Updating the network

## Troubleshooting

### Validation errors

Run the validation script from the repository root. Check that all required CSV files are present under `data/processed/` and that file names have not been changed.

### Missing CSV files

Confirm that the processed files listed in `data/metadata/data_inventory.csv` are present. If files have been renamed, update the validation script and results crosswalk.

### Unexpected manuscript values

Use `data/metadata/results_crosswalk.csv` to identify which processed output supports the manuscript claim, then inspect the corresponding CSV file.

### Spatial interpretation issues

Remember that assigned routes are straight-line planning links. They are not road routes, travel-time estimates, observed referrals, or confirmed patient pathways.

### Readiness interpretation issues

Readiness scores are based on binary service indicators. They do not confirm quality, staffing, stock reliability, blood availability, or effective coverage.
